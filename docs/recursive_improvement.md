# Recursive Improvement Loop

*Deliverable 7 / PRD §9.*

## The loop, as built

```
            slh.harness.run_population("v1")
                       │
                       ▼
  baseline = evaluate(results)        ──►  slh.report.failure_report
                       │
                       ▼
  failure clusters by persona × dimension   ←── informs the V2 design
                       │
                       ▼
  V2 policy (TutorV2 in slh/tutor.py)
                       │
                       ▼
            slh.harness.run_population("v2")
                       │
                       ▼
  improved = evaluate(results)        ──►  slh.report.failure_report
                       │
                       ▼
  slh.comparator.compare(baseline, improved)
        │   ├── per-dimension Δ (overall + per persona)
        │   ├── regression check (any dimension/persona that got worse)
        │   └── five PRD §8.2 counter-metrics, each pass/warn/fail
        ▼
  Interpretation: real improvement, or benchmark gaming?
```

This is **one complete cycle**, satisfying PRD §9.3. The loop is semi-automated
(per PRD §9.2): the harness *surfaces* the failure clusters and runs the
comparison automatically; a human authored the V2 policy informed by V1's
failure report. A future iteration could replace that human step with an LLM
proposing patches and the harness vetoing them via the same gate.

## What V1's failure report said

(See `docs/failure_report_baseline.md` — regenerated by `slh compare`.)

The failure clusters were unambiguous:
- **Tutor-Failure dimensions** all failed: answer-giving 0.82, recovery 0.05,
  shallow-compliance acceptance 1.00.
- **Curriculum-Failure dimensions** all failed: misconception persistence 1.00,
  scaffolding independence 0.10, learning gain 0.07.
- Persona clusters: `shortcut_seeker`, `adversarial_learner`, and `i_get_it`
  drove most of the tutor failures; `memorizer` exposed the transfer gap.

## What V2 changed

Six concrete policy changes (full diff in `slh/tutor.py`):

1. **Withhold answers** — V2 has no path that emits `GIVE_ANSWER`.
2. **Cap hints at 1 per item** — further requests become `PROBE`.
3. **Replace `COMPREHENSION_CHECK` with `PROBE`** — accept advance only on a
   sound `EXPLAIN`.
4. **Diagnose & remediate misconceptions** — wrong answers matching a
   misconception trigger a probe + targeted hint; `item.remediated=True` so
   `learner.resolve()` clears the misconception.
5. **Explicit `REDIRECT` and `ENCOURAGE`** for off-task and give-up.
6. **`REFUSE_BYPASS`** for adversarial attempts, with the learner's
   `bypass_blocked` flag preventing repeated attempts.

## What the comparison says

(See `docs/baseline_vs_improved.md` for the regenerated table.)

- 7/7 dimensions improved meaningfully.
- All 5 counter-metrics pass.
- Transfer gain (+0.139) **exceeds** in-lesson gain (+0.091) — the strongest
  available evidence against benchmark overfit.
- **1 regression honestly reported**: `anxious_learner /
  scaffolding_independence` — V2 probes them more, raising their hint count.
  This is a real residual finding, not a measurement artefact; see
  `docs/limitations.md`.

## How a future cycle would extend this

The harness is built for repeated cycles, not just one. To add a cycle:

1. Generate `docs/failure_report_improved.md` (already done — it shows V2's
   remaining failing dimensions).
2. Propose a V3 policy targeting those (e.g., a remedial loop for anxious
   learners that doesn't penalise their independence score).
3. Re-run `slh compare` — the comparator already supports arbitrary tutor
   names; only the harness wiring needs a new tutor.
4. The golden set in `slh/goldenset.py` becomes the regression-protection: if
   any V3 change breaks a previously-passing expectation, the gate fails.

That continuous-improvement property is why the golden set lives in code, not
prose.
